# Claude 结构化规划指南：让 AI 编程告别"自由发挥"

> 适用对象：使用 Claude Code 进行复杂任务开发的开发者
> 场景：收到模糊/复杂需求 → 需要 AI 输出可靠、可验证的代码 → 但担心 AI 自由发挥导致偏离需求
> 核心策略：**先规划后执行，用结构化流程约束 AI 行为，在每个关键节点锁定决策**
> 版本：v1.0 | 2026-06

---

## 📌 核心概述：这份文档帮你做什么

### 为什么做这件事

```
用 AI 编程，不是为了"让 AI 随便写代码"，
而是为了三件事：
```

| 目标 | 说明 |
|:-----|:------|
| 🎯 **提升代码质量** | 让 AI 输出结构清晰、逻辑一致的代码，而非东拼西凑的"能用就行" |
| ⏱ **减少返工成本** | 需求明确 + 方案确认后编码，避免"写完了发现不对"的无效循环 |
| 🧠 **驯服 AI 幻觉** | 用结构化约束让 AI 不再自由发挥，在早期锁定设计决策 |

### 要解决的核心问题（按严重程度）

```
① 需求模糊 → AI 自由发挥 → 偏离轨道
   你说了"做一个用户管理系统"，AI 就开始写代码
   → 写完发现数据库用错了、权限模型不是你想要的
   → 这是最贵的问题：花了最多时间，偏差最大

② 长上下文丢失 → AI 忘记早期决策
   聊到第 10 轮，AI 已经不记得第 2 轮定下的技术选型
   → 后面的代码与前面的设计自相矛盾
   → 需要不断"提醒"AI 之前说了什么

③ 缺乏校验节点 → 隐藏缺陷累积
   没有分阶段验收，所有问题在最后一次性爆发
   → 返工成本指数级上升
   → 什么时候出了问题都不知道
```

### 结构化规划如何解决这三个问题

```
问题 ① 指令模糊
  → 需求解析模板：强制 AI 输出结构化需求概要
  → 你确认后 AI 才能进入下一步
  → 效果：写代码前双方已达成共识

问题 ② 上下文丢失
  → 书面方案等待确认：决策被锁定在方案中
  → AI 后续所有工作围绕已确认的方案展开
  → 效果：方案就是"外部存储"，不依赖 AI 的短期记忆

问题 ③ 缺乏校验
  → 节点验收标准：每个阶段都有明确的交付物和检查点
  → 通过后才进入下一节点
  → 效果：问题在最早阶段被捕获
```

### 全貌一张表

| 阶段 | 做什么 | 产出物 |
|:-----|:-------|:-------|
| **需求解析** | 模糊需求 → 结构化概要（类型、边界、约束） | 需求结构化文档 |
| **资源审计** | 查 Agent 注册表、扫 Skill 目录、看 MCP 工具 | 能力缺口报告 |
| **可行性评估** | 5 维度评估：Agent/技能/技术栈/依赖/复杂度 | 风险清单 |
| **5 模块方案** | 目标/路径/缺口/计划/编排 → 等你确认 | 完整执行方案 |
| **等待确认** | 你确认或调整，确认后才开始编码 | ✅ 开始执行 |

## 第一章：核心策略说明

### 1.1 为什么需要结构化规划？

直接用自然语言让 AI 写代码，就像让一个顶级厨师听你描述"做一道好吃的菜"就开始做——他可能做出米其林三星，但大概率不是你想要的。

**AI 编程的天然缺陷：**

| AI 的特性 | 问题 | 后果 |
|:----------|:-----|:-----|
| 续写模式 | AI 倾向于"接着说"，而不是"先想清楚再说" | 早期决策质量低，后续代码基于错误前提 |
| 上下文窗口有限 | 聊深了会忘记早期的约定 | 代码前后矛盾，需要反复纠正 |
| 过度迎合用户 | AI 很少主动说"这个需求有问题" | 明知不合理的需求也会硬做 |
| 擅长局部优化 | 单段代码质量高，但整体架构容易松散 | 每个函数都好看，合在一起不对 |

**结构化规划的解决思路：**

```
传统模式（自由发挥）：
  需求 → AI 直接编码 → 你检查 → 修改 → 反复...
  （问题在后期集中爆发）

结构化规划模式（先规划后执行）：
  需求 → 解析 → 审计 → 评估 → 方案（等你确认） → 分节点编码 → 逐节点验收
  （问题在早期被捕获）
```

### 1.2 三条核心约束

结构化规划不是增加流程负担，而是用三条规则帮 AI 做对事：

```
规则 1：写代码前，先写方案
  → 消除需求模糊性
  → 锁定关键决策
  
规则 2：方案等你确认
  → 确认前 AI 不能写代码
  → 你的一句"不对"比 10 轮修改更高效

规则 3：分节点验收
  → 每完成一个节点，停下来检查
  → 问题发现在最早阶段
```

### 1.3 完整工作流

```
第①步：需求解析
  └─ 原始需求 → 结构化概要 → 类型判定 → 边界约束

第②步：资源审计
  └─ 查 Agent 注册表 → 扫 Skill 目录 → 看 MCP 工具 → 识别缺口

第③步：可行性评估
  └─ 5 维度评估（Agent/技能/技术栈/依赖/复杂度）+ 风险预判

第④步：5 模块方案输出
  └─ 目标理解 → 技术路径 → 缺口清单 → 工作计划 → 任务编排

第⑤步：等待确认
  └─ 你确认 → 进入标准执行流程
  └─ 你要求调整 → 修改后重新展示
  └─ 你否决 → 结束
```

## 第二章：任务分级——不同复杂度不同策略

> 不是所有任务都需要走完整的 5 步流程。分级的目的是：**重要的事不漏，简单的事不拖。**

### 2.1 四级任务定义

| 级别 | 名称 | 描述 | 例子 |
|:----:|:-----|:-----|:-----|
| **S 级** | 战略级 | 涉及多个模块、多轮迭代、架构变更 | 从零搭建一个完整项目、重构核心模块 |
| **A 级** | 复杂级 | 涉及 2+ 文件、有设计决策、需权衡 | 新增功能模块、修改数据流、对外发布物料 |
| **B 级** | 常规级 | 单文件修改、逻辑清晰、无架构影响 | 修 Bug、加个小功能、调整样式 |
| **C 级** | 简单级 | 一眼能看透、纯执行、无歧义 | 改文案、修拼写、查配置、回答问题 |

### 2.2 各级的规划深度

```
S 级 → 完整 5 步 + Agent 调度链
   第①步需求解析 → 第②步资源审计 → 第③步可行性评估 → 第④步 5 模块方案 → 第⑤步等待确认
   
A 级 → 完整 5 步（可简化审计环节）
   第①步需求解析 → 第②步资源审计（快速） → 第③步可行性评估（快速） → 第④步 5 模块方案 → 第⑤步等待确认

B 级 → 简化版：需求解析 + 简略方案 + 确认
   需求理解 → 简略方案（2-3 句话） → 你确认 → 执行

C 级 → 直接执行（报备即可）
   这类任务不需要方案，直接处理
```

### 2.3 快速判定表

```
遇到一个需求时，问自己三个问题：

Q1：涉及几个文件？
  • 1 个 → 可能是 B/C 级
  • 2-5 个 → 可能是 A/B 级
  • 5 个以上 → 可能是 S/A 级

Q2：有没有架构或设计决策？
  • 没有 → 可能是 B/C 级
  • 有，但简单 → 可能是 B 级
  • 有，需要权衡 → 可能是 A 级
  • 有，涉及整体架构 → 可能是 S 级

Q3：返工成本高不高？
  • 低（改几行就行）→ 可跳过规划
  • 中（改多个文件）→ 至少 B 级规划
  • 高（可能重构）→ 必须 S/A 级规划
```

## 第三章：需求解析——把模糊需求变成结构化输入

> "帮我做个用户管理系统"——这种需求直接让 AI 写代码，十有八九要翻车。

### 3.1 需求解析模板

当你说出一个需求时，AI 应该输出以下结构化概要：

```
① 核心需求（一句话）
   xxx

② 拆解子目标
   1. xxx
   2. xxx
   3. xxx

③ 边界与约束
   • 技术栈：xxx
   • 兼容性：xxx
   • 安全要求：xxx
   • 性能要求：xxx

④ 标记所有需用户决策的点
   • ⚠️ 需要你决定：xxx
   • ⚠️ 需要你决定：xxx
```

### 3.2 类型判定

| 类型 | 特征 | 规划方式 |
|:-----|:-----|:---------|
| **新项目** | 从零搭建，无现有代码 | 完整 5 步，需架构设计 |
| **现有项目变更** | 在已有项目上修改 | 快速审计现有代码结构后出方案 |
| **纯查询/咨询** | 不需要写代码 | 直接回答，不进入规划流程 |

### 3.3 边界约束识别

一些经常被遗漏的关键约束：

```
技术栈约束：
  • 前端用什么框架？(React / Vue / 原生)
  • 后端用什么语言？(Python / Node / Go)
  • 数据库用什么？(MySQL / PostgreSQL / SQLite)
  • 部署方式？(Docker / 传统 / Serverless)

非技术约束：
  • 时间要求：什么时候要？
  • 质量要求：是"能用就行"还是"生产级"？
  • 协作要求：别人要看/改这个代码吗？
  • 发布要求：公开还是私有？
```

### 3.4 好的 vs 不好的需求解析

**不好的需求**（直接让 AI 写代码的后果）：

```
你：帮我写个博客系统
AI：好的，开始写...

结果：AI 用了 Flask，但你想用 Django
      AI 用了 SQLite，但你想用 PostgreSQL
      AI 写了 Markdown 渲染，但你想要富文本编辑器
      → 全部重来
```

**好的需求解析**（先结构化再编码）：

```
你：帮我写个博客系统

AI 先输出：
① 核心需求：一个个人博客系统，支持文章发布和管理
② 拆解子目标：
   1. 文章 CRUD
   2. Markdown 编辑和预览
   3. 标签分类
   4. 后台管理界面
③ 需你决定的点：
   ⚠️ Python Flask vs Django？
   ⚠️ 数据库选 SQLite vs PostgreSQL？
   ⚠️ 需要用户登录系统吗？

你回答后，AI 才开始写代码。
```

## 第四章：资源审计——摸清家底再动手

> 在执行复杂任务前，先搞清楚有什么可用、缺什么。

### 4.1 三层调度架构

```
L1：Controller（唯一总控）
  └─ 不做功能性事务，只做分解调度
  └─ 收到需求 → 任务分级 → 查注册表 → 调度 L2

L2：域调度层
  ├─ Planner（产品域）→ 产品需求、文档、PRD
  ├─ Architect（研发域）→ 编码、审查、测试
  └─ Deployer（运维域）→ 部署、配置、发布

L3：执行层
  ├─ L3 由 L2 调度，不跨级调用
  └─ 每个 L3 有专属职责边界
```

### 4.2 Agent 注册表管理

所有 Agent 角色存放在 `agents/AGENT_REGISTRY.json` 中：

```json
{
  "agents": [
    {
      "id": "planner",
      "level": 2,
      "domain": "product",
      "description": "产品需求分析和文档撰写",
      "skills": ["doc-writer", "req-writer"]
    },
    {
      "id": "architect",
      "level": 2,
      "domain": "engineering",
      "description": "技术架构和代码实现",
      "skills": ["coder", "reviewer", "tester"]
    }
  ]
}
```

> **为什么要注册表？** AI 不靠"记忆中有什么 Agent"来调度，而是靠查表。注册表确保调度决策可追溯、可维护。

### 4.3 能力缺口识别

审计完后，输出格式化的缺口报告：

```
✅ 可用：
  planner L2 ─ 需求分析
  architect L2 ─ 技术实现
  doc-writer Skill ─ 文档生成
  MCP: filesystem, github

❌ 缺失：
  deployer L2 ─ 部署环节需要手动操作
  tester L3 ─ 缺少自动化测试 Agent

⚠️ 替代方案：
  部署 → 手动执行部署脚本
  测试 → 使用 pytest 手动运行
```

### 4.4 如何扩展

| 场景 | 操作 |
|:-----|:-----|
| 新增 Agent | 在 `agents/AGENT_REGISTRY.json` 添加条目，在 `agents/` 下创建对应目录 |
| 新增 Skill | 在 `skills/` 下创建，更新 CLAUDE.md 中的技能清单 |
| 注册 MCP | 在 Claude Code 配置中添加 MCP 服务器地址 |

## 第五章：可行性评估——5 维评估模型

> 在动手之前，从 5 个维度评估这个任务能不能做、怎么做、有什么风险。

### 5.1 评估维度

| 维度 | 评估内容 | 评估标准 |
|:-----|:---------|:---------|
| **Agent 完整度** | 当前注册表中是否有处理此任务的 Agent 链 | ✅ 有完整 L2→L3 链 / ⚠️ 部分缺失 / ❌ 大部分缺失 |
| **Skill 覆盖度** | 现有 Skill 是否能覆盖任务所需能力 | ✅ 全覆盖 / ⚠️ 需补充 / ❌ 需新建 Skill |
| **技术栈可行度** | 技术选型是否合理、团队是否掌握 | ✅ 成熟技术栈 / ⚠️ 需学习 / ❌ 不可行 |
| **依赖风险** | 外部 API/服务/库是否可获取 | ✅ 已有 / ⚠️ 需申请 / ❌ 不可获取 |
| **复杂度评估** | 任务本身的复杂程度 | 低/中/高（参考任务分级） |

### 5.2 风险评估格式

```
风险清单：

🔴 高风险：
  1. xxx → 影响：xxx → 缓解措施：xxx
  2. xxx → 影响：xxx → 缓解措施：xxx

🟡 中风险：
  1. xxx → 影响：xxx → 缓解措施：xxx

🟢 低风险：
  1. xxx → 不影响执行
```

### 5.3 评估结果决定后续策略

| 评估结论 | 行动 |
|:---------|:-----|
| ✅ 可行 | 进入 5 模块方案输出 |
| ⚠️ 条件可行（有风险但有缓解措施） | 进入 5 模块方案，标注风险 |
| ❌ 不可行 | 返回需求阶段，建议调整需求或补充资源 |

## 第六章：5 模块方案输出——核心交付物

> 这是整个规划流程的输出，也是你确认 AI 是否理解正确、路径是否合理的唯一依据。

### 6.1 完整模板

```
[L1] Controller | 任务分级：X 级

① 目标理解与拆分
   核心需求：xxx
   拆解子目标：
     1. xxx
     2. xxx
     3. xxx
   边界与约束：
     • xxx
     • xxx

② 技术路径与资源调度
   L2 Agent 链：xxx → xxx → xxx
   L3 SubAgent：xxx
   Skill：xxx
   MCP：xxx

③ 缺口与需协助事项
   ❌ 缺失：xxx → 影响说明
   ⚠️ 需你决策：xxx
   ⚠️ 需要你提供：xxx

④ 工作计划和节点
   节点 1：xxx
     目标：xxx
     验收标准：xxx
     交付物：xxx
     负责人：xxx
     预估工作量：xxx
   节点 2：...

⑤ 任务编排
   [串联] xxx → xxx → xxx
   [并行] xxx / xxx
   风险：xxx → 兜底：xxx
```

### 6.2 各模块详解

#### 模块 ①：目标理解与拆分

这是最重要的模块——**确保你和 AI 对"要做什么"达成一致**。

```
好的目标拆分特征：
  • 每个子目标独立可验证
  • 覆盖核心需求的全部范围
  • 没有模糊表述

不好的目标拆分特征：
  • 子目标之间重叠
  • 存在无法验证的目标（如"优化性能"——优化到什么程度？）
  • 遗漏了核心需求的关键部分
```

#### 模块 ②：技术路径与资源调度

列出具体由谁（哪个 Agent）、用什么（哪个 Skill/MCP）完成。

```
Agent 链 = 谁来干
  L2 Planner：负责需求确认和文档
  L2 Architect：负责技术实现
  L3 Coder：负责具体编码
  L3 Reviewer：负责代码审查

Skill = 用什么工具
  /docx：生成 Word 文档
  /srt-subtitle：处理字幕
  /html-ppt-maker：制作 HTML 幻灯片

MCP = 连接什么外部系统
  文件系统：读写项目文件
  GitHub：代码仓库操作
```

#### 模块 ③：缺口与需协助事项

**诚实面对短板**。缺什么、需要你提供什么，在这里一次性说清楚，避免写到一半才发现。

```
❌ 缺失 Skill：部署相关的 Skill → 部署环节需手动操作
⚠️ 需你决策：数据库选型（SQLite vs PostgreSQL）→ 影响后续所有代码
⚠️ 需你提供：API Key（调用第三方服务）
```

#### 模块 ④：工作计划和节点

每个节点的核心要素：

```
节点 = 一个可独立交付的工作单元

每个节点必须包含：
  • 目标：这个节点要完成什么
  • 验收标准：怎么判断"做完了"
  • 交付物：输出什么（代码文件、文档、配置文件）
  • 负责人：谁来做（哪个 Agent/Skill）
  • 预估工作量：大概多少行代码/多长时间
```

#### 模块 ⑤：任务编排

```
[串联] A → B → C
  B 依赖 A，C 依赖 B
  适合：有明确前后依赖关系的任务

[并行] A / B / C
  A、B、C 互不依赖，可同时进行
  适合：独立模块开发、多维度审查

[混合] (A / B) → C → (D / E)
  A 和 B 并行完成 → 合并后做 C → C 完成后做 D 和 E
  适合：大多数实际项目的真实情况

风险与兜底：
  每个风险点要有具体的兜底措施
  ❌ 不好的风险描述："可能延期"
  ✅ 好的风险描述："第三方 API 可能限流 → 兜底：使用本地 mock 数据优先开发"
```

### 6.3 完整示例

```
[L1] Controller | 任务分级：A 级

① 目标理解与拆分
   核心需求：为现有 Flask 博客系统添加 Markdown 文章导入功能
   拆解子目标：
     1. 文件上传接口（支持 .md 文件）
     2. Markdown 解析和 HTML 转换
     3. 导入预览和确认流程
     4. 导入结果展示
   边界与约束：
     • 使用现有 Flask 框架，不引入新框架
     • 数据库仍用 SQLite
     • Markdown 解析用现有 marked.js 库

② 技术路径与资源调度
   L2 Agent：architect → coder → reviewer
   Skill：无（纯代码实现）
   MCP：filesystem（操作项目文件）

③ 缺口与需协助事项
   ⚠️ 需你确认：上传文件大小限制设为多少？

④ 工作计划和节点
   节点 1：后端接口
     目标：实现文件上传和解析 API
     验收标准：POST /api/import-markdown 接受 .md 文件，返回解析后的 HTML
     交付物：routes/import.py
     负责人：L3 Coder
     预估工作量：~80 行
   节点 2：前端界面
     目标：上传界面 + 预览 + 确认
     验收标准：可选择文件、预览解析结果、确认导入
     负责人：L3 Coder
     预估工作量：~150 行
   节点 3：数据持久化
     目标：导入的文章写入数据库
     验收标准：导入后文章出现在文章列表中
     负责人：L3 Coder
     预估工作量：~50 行

⑤ 任务编排
   [串联] 节点 1（后端）→ 节点 2（前端）→ 节点 3（持久化）
   风险：marked.js 版本兼容 → 兜底：使用项目已有的版本，不升级
```

## 第七章：等待确认——反馈循环的价值

> 这是整个流程中最容易被忽略、但价值最高的环节。

### 7.1 为什么确认环节是关键

```
不确认直接做：
  10 分钟后 → AI 写完了全部代码
  你一看 → 完全不是你想要的样子
  浪费：10 分钟 AI 时间 + 你 10 分钟审查时间 + AI 重写时间

先确认再做：
  3 分钟 → AI 输出方案 → 你花 1 分钟看完
  你说"不对，数据库用 PostgreSQL"
  AI 修正方案（30 秒）
  你说"对了，开始写"
  → 后面写的代码都是对的
  节省：至少 15 分钟 + 大量情绪成本
```

**确认环节的核心价值：在写第一行代码之前，锁定所有关键决策。**

### 7.2 三种用户反馈的处理方式

| 你的反馈 | AI 的处理 |
|:---------|:----------|
| **✅ 确认** | 进入标准执行流程，按节点逐一开始 |
| **🔄 要求调整** | 修改对应模块，重新展示方案，再次等你确认 |
| **❌ 否决** | 结束流程，不执行任何编码操作 |

> **重要规则**：AI 不得在输出方案前直接编码或回答。这是硬约束，不是建议。

### 7.3 如何避免"确认疲劳"

每次都要等你确认，会不会很烦？以下情况可以跳过或简化：

```
可以跳过的场景：
  • C 级任务（改文案、修拼写）→ 直接执行
  • 你明确说了"不用确认，直接做"
  • 已经确认过的同类任务（第 2 次类似的修改）

不能跳过的场景：
  • 涉及架构决策
  • 涉及数据模型变更
  • 涉及外部服务集成
  • 涉及多人协作
  • 涉及安全性变更
```

## 执行流程（两个 Workflow 阶段）

### 前置准备：收集审计数据

Workflow 执行过程中将依次显示：
- `[L1] Controller: 需求解析与Agent分解中...`

③ 缺口与需协助事项
   ❌ 缺失 Skill：[缺失清单] → 影响/建议
   ⚠️ 需你决策：[决策点]

④ 工作计划和节点

### 第 5 步：等待确认

- 你确认 → 进入执行阶段（第 6 步）
- 你要求调整 → 重新生成或修改参数后重跑
- 你否决 → 结束

Workflow 执行过程中将依次显示：
- `[L1] Controller: 任务分级判断中...`
- `[L2] Planner: 产品域工作中...`（仅 S/A 级需要）
- `[L2] Architect: 研发域工作中...`
  - `[L3] Coder: 编码中... (节点 N)`
  - `[L3] Reviewer: 审查中... (节点 N)`
  - `[L3] Tester: 测试中... (节点 N)`
- `[L2] Deployer: 运维域工作中...`（仅 S 级需要）

## 重要规则

1. **规划阶段不编码** — pre-planning-orchestrator 只输出方案，不写代码
2. **必须等待你书面确认后才进入执行阶段**
3. **你确认后 controller-workflow 全权执行**，中间不再打断
4. B 级以下任务可简化：pre-planning 会跳过技能审计和可行性评估阶段

## 第九章：最佳实践与进阶技巧

### 9.1 Workflow 驱动的分级执行

不同任务级别走不同的 Workflow 执行链：

```
B 级简化流程：
  ① 用 2-3 句话快速确认需求理解
  ② 输出简略方案（1-2 段话）
  ③ 你确认
  ④ 执行

示例：
  你："给博客添加一个阅读量统计"

  AI："明白，给博客文章添加阅读量统计功能。
  方案：(1) 在文章数据库表加 views 字段 (2) 文章页面加载时 +1 (3) 显示在文章标题下方
  需要你确认：是否需要防刷机制？（同一个 IP 不重复计算）
  可以开始吗？"

  你："可以，不用防刷"
  AI：开始编码
```

### 9.2 与 MCP 工具配合

当任务涉及外部系统时，MCP 是规划的重要考量：

```
场景：需要读取 GitHub Issue 来规划任务
  ① 在需求解析阶段标注"需要访问 GitHub"
  ② 在资源审计阶段确认 gh CLI 或 GitHub MCP 可用
  ③ 在方案中标注"使用 GitHub MCP 读取 Issue #123"
  ④ 执行阶段自动调用 MCP 工具
```

### 9.3 多轮任务的规划策略

对于持续多天的任务，不要一次规划全部：

```
推荐方式：滚动规划
  第一天：
    规划并执行节点 1-2
    结束时快速规划明天的节点 3-4

  第二天：
    回顾昨天的结果，调整节点 3-4 的方案
    执行

  优点：
    • 计划始终基于最新状态
    • 不会出现"第一天定的计划，第三天发现不适用"
    • 每个 Session 都有明确起点和终点
```

### 9.4 规划方案的存档价值

5 模块方案不仅是指令，也是**项目文档**：

```
存档方式：
  • 每个重要任务生成的方案保存在 project-plans/ 目录
  • 文件名格式：YYYY-MM-DD-任务名.md
  • 后续回顾时可以回溯"当初为什么这么做"

存档的价值：
  • 新成员加入时快速了解项目设计思路
  • 出现争议时有据可查
  • 可以用来训练/调整 AI 的行为模式
```

## 第十章：常见问题 FAQ

### Q1：每次都要走完整 5 步，会不会太慢？

**答**：不是所有任务都需要完整 5 步。参考第二章的任务分级——S/A 级走完整流程，B 级简化，C 级直接执行。**复杂任务慢下来，是为了整体快起来**。一个错误的架构决策导致的重构成本，远大于规划阶段的 5 分钟。

### Q2：AI 输出的方案跟我想要的不一样，怎么办？

**答**：直接告诉 AI 哪里不对。方案只是一个草案，你的调整指令是规划流程的**关键输入**。好的分工是：**你决定方向和关键决策，AI 负责填充细节和执行。**

### Q3：Agent 注册表怎么维护？

**答**：
- 新建 Agent → 在 `agents/AGENT_REGISTRY.json` 中添加条目 + 创建对应目录
- 废弃 Agent → 从注册表中移除
- 定期审查 → 每次重要任务前快速过一遍注册表，确保调度链完整
- 简化方案 → 小团队可以不用 Agent 注册表，直接用 Skill 清单做资源审计

### Q4：和直接让 Claude 写代码有什么区别？

**答**：直接让 Claude 写代码 = **让一个天才在黑暗中乱跑**；结构化规划 = **先点亮所有灯，再让他跑**。

| 对比项 | 直接写代码 | 结构化规划 |
|:-------|:-----------|:-----------|
| 需求理解 | 靠 AI 从上下文中猜测 | 输出结构化概要让你确认 |
| 技术选型 | AI 默认选择 | 方案中列出选项由你决定 |
| 执行路径 | 一次性全部完成 | 分节点逐一验收 |
| 出错时机 | 最后才发现 | 每个节点前锁定 |
| 可追溯性 | 聊完就没了 | 方案可存档 |

### Q5：这个方法论能用于非 Claude 的 AI 工具吗？

**答**：可以。这套方法论的核心是 **"先规划后执行"和"分阶段确认"**，不依赖特定的 AI 工具。你可以在 ChatGPT、Cursor、GitHub Copilot 等任何 AI 编程工具中使用相同的思路——**关键不是工具，而是流程**。

### Q6：如果我忘了触发 /plan-action，直接说了需求怎么办？

**答**：你会得到一次"非结构化"的体验——然后就能体会到为什么需要规划。**任何时候都可以喊停，说"用 /plan-action 重新规划这个任务"**，AI 会重新输出结构化方案。

### Q7：规划方案存档多了会不会乱？

**答**：
- 按年月组织目录：`project-plans/2026/06/`
- 文件名包含任务关键词
- 过期的方案可以归档到一个 `_archive` 子目录
- 不存档 C 级任务方案（没意义）
